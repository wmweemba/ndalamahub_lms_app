Investigated the phase doc's pointer at "the existing `latePayment`/penalty enums" — the actual fields are `gracePeriod.interestDuring` (`['none','accrued','capitalized']`) and `prepayment.penalty`/`penaltyRate`; grepped and confirmed both are referenced only in `utils/seedProducts.js` seed data, nowhere in route/model logic — marked dormant in a comment, left intact (they're a different, unrelated concept — moratorium/early-payoff penalty — not the rollover mechanic; not repurposed). `CreateProductDialog.jsx`/`EditProductDialog.jsx` both gained an additive "Rollover" section (enable checkbox + grace-days input); payload shape is purely additive, both dialogs already spread `formData` into the POST/PUT body.

New `rollover()` instance method: sums outstanding across every not-paid/waived/rolled installment (partial payments already reduce this via `paidAmount`), capitalizes it as the new principal, applies fresh interest at the loan's own `interestRate`, anchors the new due date to the *previous* due date (`addMonths`/`addDays` per `termUnit`, not "today + term"), marks the superseded installment(s) `'rolled'`, pushes one fresh `'pending'` installment for the new total due, and returns the loan to `'active'`. Throws on `defaulted` or any terminal/pre-disbursement status. Audited and updated every place that special-cases `'waived'` to also exclude `'rolled'`: `calculateAccruedInterest()` and `calculateEarlySettlementAmount()`'s remaining-scheduled-interest loop (both — otherwise a rolled installment's interest would be double-charged, since it's already folded into the new capitalized principal). `getSummary()`'s `remainingBalance` and `calculateRemainingBalance()` (used by prepayment/settlement) both branch on `rolloverCount > 0`: post-rollover, the original `amount`/`totalAmount` no longer reflect what's owed (deliberately never mutated — disbursement reporting must stay accurate), so current exposure is computed off the live (non-paid/waived/rolled) schedule entries instead. Verified via a settlement-after-rollover unit test: quoting immediately after a rollover returns exactly the capitalized principal with zero accrued interest (the new installment isn't due yet), confirming the "no charge during grace" mechanic holds through the settlement path too.
- **Step 3 (the job)**: `jobs/rolloverLoans.js` — finds loans on rollover-enabled products, status `active`/`in_arrears`, with outstanding balance, whose current due date is strictly past `dueDate + product.rollover.graceDays`; calls `loan.rollover()` sequentially (model hooks stay in the loop, per the Phase 07 precedent) and emails the borrower if they have one (`emailTemplates.loanRolledOver()`, via the existing never-throws `sendEmail`). Idempotent by construction — a loan with no outstanding, non-rolled installment past its grace window is simply not selected on a second run. Fixed with the exact pattern already established in `sendPaymentReminders.js`: `require('../models/User'); // registers the schema populate('applicant', ...) needs`.
- **Step 4 (notifications & manual default)**: `emailTemplates.loanRolledOver(user, loan, rolloverEntry)` added (same `layout()`/`formatMoney()`/`formatDate()` helpers as the existing loan templates). New `PUT /api/loans/:id/default` (`routes/loans.js`) — `authorize('lender_admin')` at the route (mirrors `/disburse`'s existing "lender_admin only, no platform_admin" pattern) plus a `canWriteRepayment()` tenancy check in-handler; requires `reason` in the body (400 without one), 400 if already defaulted, appends a note, sets `status = 'defaulted'`.
